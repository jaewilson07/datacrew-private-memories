## Active Opportunity: AI Chief of Staff for CFOs (2026-07-13)

Frederick DeWorken asked Jae directly: "How confident would you feel delivering an AI-driven Chief of Staff for a CFO using the Anthropic + DOMO toolset? What approach would you take?"

This is a live opportunity. Frederick is interested in delivering this as a BlueYeti offering. Domo is for visualization. The research we've done on the AI CFO project maps directly to what Frederick is asking.

### Key research findings that apply:
- Anthropic's own finance team built 150 Claude Skills (MFR, variance, reconciliation, board packs)
- The winning pattern is a **skill library with verification**, not a chatbot
- The "checking layer IS the product" — every output needs source citations, exception flags, reconciliation tabs
- Domo's role = data foundation + visualization layer (governed datasets, row-level security, certified data)
- The 4-part prompt framework: Context, Input format, Output spec, Exception handling

### What we need to investigate:
1. Domo MCP server capabilities — can Claude call Domo cards/datasets directly?
2. Domo Agent Catalyst — current state
3. Domo's API for programmatic card/dashboard creation
4. What BlueYeti has already built (Snowflake-native agents, Domo agentic workflows)
5. BlueYeti's delivery model for AI projects
6. What Frederick means by "Chief of Staff" (chatbot vs skill library vs proactive agent)

### Recommended approach:
Build a skill library on top of Domo's data layer — not a chatbot. 5 layers: data foundation (Domo), skill layer (Claude), verification layer, presentation layer (Domo dashboards), governance layer.

### Evolved architecture (2026-07-13 design session):
The AI CoS design evolved into a 5-layer "Watchtower" pattern with Domo Goals/OKRs as the context layer. Key insights from the design session:

- **The output is a daily brief, not a dashboard** — a Slack message with "click into it" links to underlying data, Jira tickets, Domo cards with filters applied
- **Domo Goals/OKRs are the context layer** — the CoS reads OKRs to know "what matters," filters to At Risk + Needs Attention, pulls underlying card data, synthesizes into brief
- **Domo Semantic Layer + Lineage** enables tracing from OKR metric → card → dataset → source connector, identifying WHERE breaks occur
- **"Dark data" concept** — metrics not tracked or not updated = areas needing attention (process/people problem, not just technology)
- **Key insight from Jae: "Step one is good OKRs"** — the consulting engagement IS the foundation work (set up OKRs, link to Domo cards, use lineage for diagnostics). The AI CoS layer comes after the foundation is solid. This is a people/technology/process problem, not just technology.
